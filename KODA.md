# KODA.md — Инструкции для проекта HardWorker

## Обзор проекта

**HardWorker** — это десктопное приложение для мониторинга и управления промышленным оборудованием. Приложение работает под Windows и предоставляет графический интерфейс для взаимодействия с различными устройствами (сепараторы, вискозиметры).

### Основные технологии

- **Язык**: Go 1.25.6
- **GUI-фреймворк**: giu (обёртка над imgui)
- **Системный трей**: github.com/getlantern/systray
- **Протокол связи**: Modbus (github.com/simonvetter/modbus)
- **gRPC**: google.golang.org/grpc + google.golang.org/protobuf
- **LabVIEW**: Проект для прошивки оборудования (директория lv/)

### Архитектура

```
HardWorker/
├── main.go              # Точка входа, инициализация модулей
├── grpc.proto           # Определение gRPC сервиса
├── Makefile             # Скрипты сборки
├── assets/              # Ресурсы (иконки, шрифты)
├── pkg/
│   ├── app/             # Ядро приложения (события, состояние)
│   ├── akip/            # Модуль "Сепаратор Ультразвуковой"
│   ├── visko/           # Модуль "Вискозиметр Магнитный"
│   ├── setts/           # Модуль настроек
│   ├── loger/           # Модуль логирования
│   ├── tray/            # Управление системным треем
│   ├── ui/              # Графический интерфейс
│   └── proto/           # Сгенерированный gRPC код
└── lv/                  # LabVIEW проект (прошивка оборудования)
```

### Подключаемое оборудование

В текущей конфигурации (main.go) активирован модуль:
- **AKIP**: Сепаратор Ультразвуковой (адрес: 192.168.0.100:3000, gRPC порт: 50051)

Отключённые модули (закомментированы):
- **Visko**: Вискозиметр Магнитный (192.168.0.200:502)

## Сборка и запуск

### Требования

- Go 1.25+
- Windows (для сборки с иконкой)
- Компилятор ресурсов `windres` (входит в MinGW)
- protobuf компилятор (`protoc`) для генерации gRPC кода

### Команды Makefile

| Команда | Описание |
|---------|----------|
| `make run` | Запуск приложения в режиме разработки |
| `make build` | Сборка исполняемого файла HardWorker.exe с иконкой |
| `make clean` | Удаление собранного exe-файла |
| `make proto` | Генерация gRPC кода из grpc.proto |

### Ручная сборка

```bash
# Обычная сборка
go build -ldflags "-s -w -H=windowsgui -extldflags=-static" -o HardWorker.exe .

# Запуск
go run .
```

### Генерация gRPC кода

```bash
protoc --go_out=./pkg/proto --go_opt=paths=source_relative \
       --go-grpc_out=./pkg/proto --go-grpc_opt=paths=source_relative \
       grpc.proto
```

### Установка отладчика (опционально)

```bash
go install github.com/go-delve/delve/cmd/dlv@latest
```

## Структура кода

### Модульная система

Приложение использует интерфейс `Modules` (pkg/app/main.go):

```go
type Modules interface {
    giu.Widget
    Run()
    Name() string
}
```

Каждый модуль (akip, visko, setts, loger) реализует этот интерфейс и предоставляет:
- `Run()` — запуск горутины для работы с устройством
- `Name()` — имя модуля для отображения в UI
- Реализацию `giu.Widget` для рендеринга в GUI

### Обработка событий

Приложение использует канал событий для коммуникации между компонентами:

```go
var Event = make(chan Events, 2)

const (
    EventToggleGUI Events = iota  // Показать/скрыть окно
    EventCloseGUI                  // Закрыть окно
    EventQuit                      # Выйти из приложения
)
```

### Проверка запущенных экземпляров

Приложение проверяет, запущен ли уже экземпляр, через TCP-слушатель на порту 127.0.0.1:54424. Если порт занят — приложение завершается.

### Системный трей

Иконка трея создаётся из внедрённого ресурса (embed директива в main.go). При клике на иконку отправляется событие `EventToggleGUI`.

## gRPC API

### Сервис Akip

Определён в grpc.proto:

```protobuf
service Akip {
  rpc Data (AkipRequest) returns (AkipResponse);
}

message AkipRequest {}
message AkipResponse {
  float level = 1;
}
```

Порт по умолчанию: `:50051`

## Особенности разработки

### Стиль кода

- Стандартные соглашения Go (camelCase для переменных, PascalCase для экспортируемых)
- Использование контекста (`context.Context`) для управления жизненным циклом
- Синхронизация через `sync.WaitGroup`
