# HardWorker Project Context

## Назначение

Это единственный канонический не-кодовый документ проекта.

Его задача:

- быстро вводить в контекст в новом thread;
- хранить текущую архитектурную правду;
- фиксировать повторно используемые инженерные решения;
- не допускать повторного обсуждения уже решённых проблем.

## Жесткая иерархия контекста

Порядок доверия такой:

1. `docs/project-context.md`
2. актуальный код
3. `AGENTS.md` как короткая инструкция для ассистента
4. `README.md` как краткая верхнеуровневая навигация

Правило:

- в `docs/` не должно быть набора параллельных source-of-truth документов;
- если появляется новое устойчивое знание, оно вносится сюда;
- если информация устарела, она исправляется здесь, а не дублируется в новом файле;
- если знание одноразовое или локальное, его не надо тащить в документацию.

## Обязательное правило для каждой существенной задачи

Если в ходе работы появилась хотя бы одна из вещей ниже, этот файл надо обновить в том же change set:

- новая архитектурная договоренность;
- новая методика реализации;
- новый повторяющийся баг и его правильное решение;
- новый workflow сборки, релиза или отладки;
- изменение зон ответственности файлов или приложений;
- новое приложение в `apps/`;
- изменение общего UI/UX правила;
- изменение контрактов между backend и frontend.

Если задача изменила проект, но этот файл не обновлён, задача считается закрытой не полностью.

## Правило актуальности документа

Этот файл должен отражать текущее состояние проекта после каждого существенного изменения.

Жёсткое правило:

- если меняется проект, должен меняться и `docs/project-context.md`;
- файл должен быть актуален после каждого комита, который меняет архитектуру, правила работы, release flow, методики, общие багфиксы или карту приложений;
- нельзя "накопить изменения и обновить документацию потом";
- нельзя терять поток развития проекта между thread и между комитами.

Практический смысл:

- новый thread должен стартовать не с восстановления истории по коду и переписке, а с уже актуальной опоры;
- если решение оказалось полезным второй раз, оно уже должно быть зафиксировано здесь;
- если правило не попало сюда, значит знание ещё не закреплено на уровне проекта.

## Как стартовать новый thread

Ожидаемый стартовый контекст:

- считать активной разработкой Wails-направление в `apps/`;
- не начинать с широкого сканирования всего репозитория;
- сначала читать этот файл;
- затем читать только те файлы, которые относятся к конкретной задаче.

Рекомендуемый формат первого сообщения пользователя:

```text
HardWorker. Смотри docs/project-context.md.
Область: apps/akip | apps/visco | apps/updater | apps/configmaster | cross-project.
Задача: ...
Не трогать: ...
Проверка результата: ...
```

## Текущее состояние проекта

Текущая активная разработка ведётся в `apps/`.

Основные продуктовые приложения:

- `apps/akip` - основной Wails desktop для AKIP;
- `apps/visco` - основной Wails desktop для VISCO.
- `apps/rp40` - desktop-калькулятор газопроницаемости по методу RP40.

Сервисные и инфраструктурные приложения:

- `apps/configmaster` - генерация и редактирование мастер-конфига update toolchain;
- `apps/updater` - клиент обновления для конечной машины;
- `cmd/release-publish` - публикация релизных артефактов и обновление `manifest.json`.

Legacy-код в корне репозитория существует, но не считается текущим target по умолчанию:

- `main.go`
- `pkg/*`
- `lv/*`
- корневой `grpc.proto`

Если задача явно не говорит про legacy, работать надо в `apps/*`.

## Общая архитектурная модель

Все активные desktop-приложения строятся по общей схеме:

- backend на `Go`;
- desktop shell на `Wails`;
- frontend на `React + TypeScript`;
- графики на `ECharts`;
- state во frontend хранится тонко, без критичной бизнес-логики.

Главное правило:

- backend является source of truth;
- frontend показывает snapshot и отправляет команды;
- вычисления, обмен с приборами, persistence и ключевые состояния живут в Go.

## Общие инженерные правила

- Нельзя переносить критичную доменную логику только во frontend.
- Если значение влияет на KPI или отображаемую физику процесса, оно должно считаться в Go.
- Frontend должен быть восстанавливаемым из snapshot.
- Дефолты, важные для оборудования, должны быть согласованы между backend и frontend fallback.
- Логи являются частью продукта, а не только отладкой.
- Ресурсоёмкие сборки и релизные действия запускать только после явного запроса пользователя.

- экспорт CSV;

Ключевые backend-файлы:

- `apps/visco/app.go`

Ключевые frontend-файлы:

### RP40

Назначение:

- загрузка паспортного tab-separated `.xls/.csv` файла, сохранённого из LabVIEW;
- выделение уникальных образцов из колонки `№ образца`;
- автозаполнение геометрии и паспортных параметров выбранного образца;
- загрузка отдельного файла замера кривой падения давления;
- расчёт газопроницаемости через внешний модуль `gcalc/pkg/RP40`;
- показ display-результата, двух ветвей расчёта и диагностических графиков.

Ключевые backend-файлы:

- `apps/rp40/app.go`
- `apps/rp40/rp40_service.go`

Ключевые frontend-файлы:

- `apps/rp40/frontend/src/App.tsx`
- `apps/rp40/frontend/src/App.scss`

Ключевые доменные правила:

- паспортный файл нужно читать как текстовый tab-separated экспорт, а не как настоящий бинарный Excel;
- типовая кодировка паспортных файлов `Windows-1251`;
- `Vt` не берётся из паспорта и задаётся пользователем через фиксированный список `17.3 / 233 / 1445 / 4876 / Custom`;
- модуль расчёта RP40 подключается как обычная внешняя Go-зависимость `github.com/pioruner/gcalc`;
- backend остаётся источником правды для парсинга паспорта, выбора параметров и запуска расчёта.
- при загрузке файла замера нужно пытаться автоматически выбрать образец по началу имени файла; если совпадения нет, оператор выбирает образец вручную.
- для `select` в Wails/WebView2 нужно явно задавать тёмный фон и цвет `option`, иначе на Windows системный dropdown может открываться светлым и делать текст нечитаемым.

## Release / update flow

Рабочая цепочка такая:

1. `ConfigMaster` хранит и редактирует мастер-конфиг.
2. `ConfigMaster` генерирует:
   - `configmaster.local.json`
   - `uploader.local.json`
   - `updater.local.json`
3. `release-publish` публикует zip или бинарный артефакт и обновляет `manifest.json`.
4. `Updater` читает `manifest.json`, сравнивает версии, скачивает релиз, проверяет `sha256`, устанавливает и запускает приложение.

Принцип:

- конечный пользователь не должен видеть инфраструктурные настройки;
- секреты не должны всплывать в обычном клиентском UI;
- release flow должен масштабироваться на несколько приложений.

## Режимы работы: локальная разработка и пусконаладка

Нужно жёстко различать два режима.

### 1. Обычная разработка новой фичи, программы или внутреннего багфикса

Разрешённый цикл:

- менять код;
- локально собирать;
- локально запускать тестовый вариант;
- проверять поведение на dev-сборке без обязательной публикации в облако.

Типичные команды:

- `make akip-dev`
- `make akip-build`
- `make visco-dev`
- `make visco-build`
- `make rp40-dev`
- `make rp40-build`
- `make rp40-release-windows`

Принцип:

- если задача ещё находится в режиме обычной разработки и не тестируется заказчиком на реальном железе через `Updater`, достаточно локальной тестовой сборки.

### 2. Пусконаладочные работы у заказчика на реальном железе

Здесь действует другое правило.

Если сборка нужна для проверки у заказчика через реальную цепочку обновления, то каждая такая сборка должна идти как релиз в облако.

Обязательный цикл:

- собрать целевое приложение под нужную платформу;
- опубликовать артефакт и обновить `manifest.json`;
- проверять установку через `Updater`, а не ручной подменой бинарников;
- тестировать именно тот путь, которым пользуется заказчик.

Принцип:

- если идёт пусконаладка, "тестовая локальная сборка" не считается достаточной проверкой;
- в этом режиме каждая проверяемая сборка должна быть доступна через наш release/update pipeline.

## Подробная инструкция по release/update pipeline

Это рабочий регламент для использования нашего средства выгрузки и обновления приложений.

### Основные сущности

- `ConfigMaster` подготавливает конфиги;
- `release-publish` публикует релиз;
- `Updater` устанавливает релиз на машине заказчика;
- `manifest.json` является точкой правды по доступным версиям;
- артефакт проверяется по `sha256`.

### Какие конфиги используются

Главные локальные файлы:

- `configmaster.local.json`
- `uploader.local.json`
- `updater.local.json`

Смысл:

- `ConfigMaster` редактирует мастер-конфиг;
- из него генерируются `uploader.local.json` и `updater.local.json`;
- `release-publish` использует `uploader.local.json`;
- `Updater` использует `updater.local.json`.

### Где искать конфиги

`ConfigMaster` работает с `configmaster.local.json`.

`release-publish` ищет `uploader.local.json` через:

- явный `-config`;
- либо стандартный lookup через `HARDWORKER_UPLOADER_CONFIG`.

`Updater` ищет `updater.local.json` по цепочке:

1. `HARDWORKER_UPDATER_CONFIG`
2. `./updater.local.json` рядом с исполняемым файлом
3. `./config/updater.local.json`
4. `workspace config/updater.local.json`

### Базовый сценарий настройки

1. Открыть `ConfigMaster`.
2. Заполнить:
   - `manifest_url`
   - storage credentials
   - список приложений
   - пути для генерации `uploader.local.json` и `updater.local.json`
3. Сохранить мастер-конфиг.
4. Сгенерировать `uploader.local.json` и `updater.local.json`.
5. Убедиться, что у `Updater` и `release-publish` доступны свои конфиги.

### Что делает release-publish

`release-publish`:

- принимает папку сборки, `.exe` или готовый `.zip`;
- если нужно, сам делает zip;
- считает `sha256`;
- загружает артефакт в object storage;
- обновляет `manifest.json` для нужного `app/platform/arch`.

### Базовая команда публикации

```bash
go run ./cmd/release-publish \
  -config ./config/uploader.local.json \
  -app akip \
  -version 2026.03.11.2 \
  -platform windows \
  -arch amd64 \
  -source ./apps/akip/build/bin/akip-wails-prototype.exe \
  -notes "AKIP release 2026.03.11.2"
```

Ключевые флаги:

- `-config` - путь к `uploader.local.json`
- `-app` - логический ID приложения в manifest
- `-version` - версия релиза
- `-platform` - обычно `windows`
- `-arch` - обычно `amd64`
- `-source` - папка сборки, `.exe` или `.zip`
- `-notes` - заметка о релизе

### Предпочтительный путь публикации в HardWorker

Обычно не нужно руками вызывать `release-publish`, если подходит `Makefile`.

Стандартные команды:

- `make akip-release-windows`
- `make visco-release-windows`
- `make windows-release`

Что они делают:

- собирают `windows/amd64` бинарник;
- вычисляют новую версию в формате `YYYY.MM.DD.N`, если `VERSION` не задан явно;
- публикуют артефакт через `go run ./cmd/release-publish`;
- обновляют локальный файл версии только после успешной публикации.

Файлы версий лежат в:

- `release/versions/akip-windows-amd64.version`
- `release/versions/visco-windows-amd64.version`

Текущий дефолтный путь к uploader config в `Makefile`:

- `release/windows-toolchain-prefilled-cloudru-20260311/uploader/uploader.local.json`

Если нужен другой конфиг, его надо передать через `UPLOADER_CONFIG=...`.

Пример:

```bash
make akip-release-windows UPLOADER_CONFIG=./config/uploader.local.json
```

### Как проверять релизный сценарий правильно

Если задача относится к пусконаладке или проверке у заказчика:

1. собрать релизную версию;
2. опубликовать её в облако;
3. убедиться, что `manifest.json` обновился;
4. на клиентской машине открыть `Updater`;
5. нажать проверку версии;
6. выполнить обновление;
7. запускать приложение уже через обновлённую поставку.

Запрещённый shortcut для пусконаладки:

- передавать заказчику бинарник вручную вместо публикации;
- тестировать установку вне `Updater`, если цель именно проверка боевого сценария обновления.

### Как понимать успешность публикации

Успешный `release-publish` должен выдать:

- `app id`
- `version`
- итоговый URL артефакта
- `sha256`
- подтверждение обновления `manifest.json`

Без этого релизный шаг нельзя считать завершённым.

### Как работать, если меняется release/update методика

Если в ходе работы меняется хотя бы одно из ниже:

- порядок генерации конфигов;
- способ версионирования;
- команды публикации;
- путь поиска конфигов;
- поведение `Updater`;
- формат или логика `manifest.json`;
- способ клиентской проверки на железе;

то изменения надо сразу внести в этот файл в том же наборе правок.

## Правила UI и графиков

Это обязательные методики, потому что они уже всплывали повторно и не должны переобсуждаться в каждом новом thread.

### Инициализация графиков

Нельзя рассчитывать, что график корректно займёт контейнер после первого рендера сам по себе.

Обязательные приёмы:

- после первого показа экрана делать принудительный `chart.resize()`;
- после переключения вкладок делать повторную стабилизацию размеров;
- использовать `requestAnimationFrame` и несколько delayed resize;
- использовать `ResizeObserver` для контейнера графика.

Причина:

- без этого `ECharts` может занять только часть контейнера до первого ручного repaint.

### Масштаб по оси Y

По умолчанию запрещено жёстко прибивать начало `OY` к `0`, если это не является осознанным доменным требованием.

Стандартное правило:

- брать фактический `min/max` по данным;
- добавлять сверху и снизу padding;
- если диапазон нулевой, добавлять технический отступ, чтобы ось не схлопывалась.

### Формат подписей

Подписи осей и графических значений должны быть доменно-осмысленными и короткими.

Стандарт:

- `T1/T2` и подобные счётчики времени отображать без длинной дробной части;
- `U*` и `Temp` ограничивать максимум двумя знаками после запятой;
- запрещены длинные хвосты floating point, которые ломают layout.

### Распределение пространства

Если на экране одновременно есть графики, таблица и метрики, графическая зона всё равно должна получать максимум полезной высоты.

Для VISCO предпочтительный принцип:

- слева таблица и summary;
- справа графики и нижняя панель курсора.

## Правила логирования

Для продуктовых приборных приложений лог-модуль обязателен.

Минимальные требования:

- источник логов находится в backend;
- хранение через ring buffer или ограниченный буфер;
- уровни: `INFO`, `WARN`, `ERROR`, `DEBUG`;
- логи доступны прямо из UI;
- UI должен поддерживать паузу обновления ленты;
- UI должен поддерживать очистку только видимой ленты без разрушения backend-потока логирования.

Что обязательно логировать:

- старт и останов сервиса;
- подключения и отключения прибора;
- ошибки чтения, парсинга, записи и сетевого обмена;
- критичные действия пользователя;
- запуск и ошибки gRPC, если gRPC используется.

## Правила расширения проекта

Проект будет расширяться, поэтому новые приложения должны добавляться без размывания общей модели.

При добавлении нового приложения в `apps/` нужно сразу зафиксировать в этом файле:

- назначение приложения;
- backend source of truth;
- ключевые entrypoints;
- пользовательские сценарии;
- общие и специфические инженерные правила;
- место в release/update flow, если оно участвует в поставке.

Нельзя добавлять новое приложение так, чтобы знание о нём осталось только в коде и в переписке.

## Что считать устареванием контекста

Контекст считается устаревшим, если уже произошло хотя бы одно из событий:

- поменялась основная архитектура приложения;
- появились новые общие UX-правила;
- найден повторяющийся баг и утверждён стандартный fix;
- поменялась схема релиза;
- поменялись активные приложения проекта;
- поменялись основные команды сборки;
- изменилась зона ответственности legacy-кода и `apps/*`.

## Практическое правило для работы со мной

Если начинаем новый thread, не нужно заново пересказывать историю проекта.

Достаточно указать:

- что задача относится к `HardWorker`;
- что опорный контекст находится в `docs/project-context.md`;
- какую область проекта трогаем;
- какие части не трогать.

Этого достаточно, чтобы быстро войти в работу без повторного широкого разгона.
