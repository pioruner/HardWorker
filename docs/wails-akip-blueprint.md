# Wails AKIP Blueprint

## 1. Цель документа
Этот документ фиксирует рабочий шаблон приложения уровня SCADA/приборной панели на `Go + Wails + React`.
Он нужен как единая точка синхронизации для новых проектов, чтобы не повторять архитектурные обсуждения в чате.

## 2. Технологический стек
- `Go 1.23+` для backend-логики, сетевого обмена, расчетов, persistence.
- `Wails v2` как кроссплатформенная desktop-обвязка (`Windows/macOS/Linux`).
- `React + TypeScript + Vite` для UI.
- `SCSS` + CSS variables для темизации и масштабируемых стилей.
- `Zustand` для frontend state.
- `ECharts` (`echarts-for-react`) для осциллограмм/трендов.
- `gRPC` (опционально) для внешнего доступа к данным приложения.

## 3. Архитектурный принцип
- Backend в Go является **источником истины**: связь с прибором, парсинг, расчет, состояние.
- Frontend не хранит критичную бизнес-логику; он показывает snapshot и отправляет контролы.
- Обмен между UI и Go через Wails bindings:
  - `GetSnapshot()` -> полный текущий state;
  - `ApplyControls(...)` -> применить изменения и вернуть актуальный state;
  - отдельные команды (`SetRegistration`, `ZeroVolumeReference`) для явных действий.

## 4. Слои приложения
### 4.1 Backend (Go)
- `AkipService`:
  - TCP reconnect loop;
  - polling прибора (waveform);
  - бинарный unpack;
  - расчет метрик;
  - автопоиск (включается только пользователем);
  - CSV-регистрация;
  - gRPC-сервер.
- Persistence:
  - load/save JSON в `UserConfigDir()/HardWorker/<service>.json`.
- Runtime lifecycle:
  - `Start(ctx)` запускает фоновые циклы;
  - `Shutdown()` корректно закрывает файлы/соединения.

### 4.2 Frontend (React)
- Store (`zustand`) хранит snapshot + локальные патчи.
- Polling `GetSnapshot()` (интервал ~250ms) для актуализации UI.
- Любое изменение контролов -> `ApplyControls(...)`.
- График строится из `snapshot.x/y`, курсоры и параметры из snapshot.

## 5. Контракт данных (рекомендованный)
- Snapshot должен включать:
  - статус связи, последний ответ, адрес;
  - параметры (`timeBase`, `hOffset`, `reper`, `square`, `minY`, `minMove`, `autoSearch`);
  - курсоры (`cursorMode`, `cursorPos`);
  - waveform (`x`, `y`);
  - расчетные поля (`vSpeed`, `vTime`, `volume`);
  - состояние регистрации.
- Controls должны включать только пользовательские настройки, без служебных полей.

## 6. Принципы UX/визуального стиля
- Плотный приборный layout: верхняя панель параметров, метрики, график, нижняя панель управления.
- Главный фокус интерфейса — осциллограмма и 1-2 ключевых KPI.
- Строка последнего ответа прибора всегда отдельной строкой с `ellipsis`.
- Стабильный темный технический стиль:
  - единые CSS variables;
  - акцентные цвета для статусов/курсоров;
  - минималистичные элементы управления.
- Окно:
  - с рамками (`Frameless: false`);
  - старт развернутым (`Maximised`) по необходимости;
  - `MinWidth/MinHeight` для защиты от развала верстки.

## 7. Принципы измерений (важно)
- Для подобных УЗ задач основной KPI обычно не абсолютный объем, а **изменение**.
- Обязательная функция: `Zero/Set reference` (обнуление перед экспериментом).
- Хранить:
  - `volumeAbs` — абсолютно рассчитанный объем;
  - `volumeRef` — опорная точка;
  - UI показывает `delta = volumeAbs - volumeRef`.

## 8. CSV-регистрация
- Запуск по кнопке пользователя (через Save dialog).
- Формат: `Date-Time, Volume ml` (или доменный формат проекта).
- Интервал записи (в текущем решении: 1 сек).
- Корректное закрытие writer/file при stop/shutdown.

## 9. gRPC-подход
- Запуск как отдельный фоновый сервер внутри приложения (например `:50051`).
- Метод `Data()` должен возвращать **актуальные вычисленные данные**, не константы.
- Нельзя блокировать UI или приборный цикл внутри gRPC-обработчиков.

## 10. Встроенный лог-модуль (обязательный)
- Логи должны быть встроены в каждое приложение и доступны прямо из UI (вкладка/экран `Логи`).
- Источник логов: backend (`Go`) через буфер фиксированного размера (`ring`/`limit`).
- Минимальные уровни: `INFO`, `WARN`, `ERROR`, `DEBUG`.
- UI-оформление:
  - цветовая кодировка по уровню;
  - новые записи сверху;
  - timestamp + level + message в каждой строке;
  - прокручиваемый список без влияния на layout основного экрана.
  - обязательная кнопка `Пауза/Продолжить`, которая замораживает обновление UI-ленты логов для чтения ошибок (backend-логирование при этом продолжается).
  - обязательная кнопка `Очистить`, которая очищает только UI-ленту (без удаления backend-истории).
- Что обязательно логировать:
  - старт/останов сервиса;
  - подключения/отключения прибора и причины ошибок;
  - критичные пользовательские действия (регистрация, zero reference, смена адреса/режима);
  - ошибки чтения/парсинга/отправки команд;
  - запуск/ошибки gRPC.
- Контракт Wails:
  - `GetLogs()` возвращает список логов для polling/UI.

## 11. Старт нового похожего проекта (чеклист)
1. Инициализировать Wails + React/TS шаблон.
2. Подключить `zustand`, `scss`, `echarts`.
3. Реализовать `Service` в Go с `Start/Shutdown/GetSnapshot/ApplyControls`.
4. Подключить polling snapshot и двусторонний binding.
5. Добавить persistence JSON.
6. Добавить регистрацию CSV.
7. Добавить `Zero reference` для дельта-метрик.
8. Добавить gRPC для внешнего чтения ключевых показателей.
9. Добавить встроенный лог-модуль (`GetLogs`, цветной UI, ключевые события).
10. Зафиксировать `MinWidth/MinHeight`, финально проверить UX на маленьких и больших окнах.
11. Собрать `windows/amd64`, `darwin`, `linux` и провести тест на реальном железе.

## 12. Ключевые инженерные договоренности
- Никакой критичной логики только во frontend.
- Любое вычисление, влияющее на KPI, должно жить в Go.
- UI должен быть “тонким”, предсказуемым и быстро восстанавливаемым из snapshot.
- Любые дефолты оборудования должны быть явно заданы и одинаковы в backend и frontend fallback.
- Ресурсоемкие сборки (`wails build`, release-артефакты) выполняются только после явного одобрения пользователя.

## 13. Где смотреть рабочую реализацию
- Backend сервис: `apps/akip/akip_service.go`
- gRPC: `apps/akip/grpc.go`
- Wails app bindings: `apps/akip/app.go`
- UI экран: `apps/akip/frontend/src/App.tsx`
- UI store: `apps/akip/frontend/src/store/akipStore.ts`
- Стили: `apps/akip/frontend/src/App.scss`

## 14. Применение к VISCO
- Отдельное приложение VISCO находится в `apps/visco` и использует ту же базовую схему:
  - backend service в Go как source of truth;
  - frontend на React/TS через Wails bindings;
  - встроенный лог-модуль (`GetLogs`, вкладка `Логи`, `Пауза/Продолжить`, `Очистить`).
- gRPC для VISCO пока не обязателен (опциональный слой из раздела 9).
